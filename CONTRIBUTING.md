# Contributing to Grainy

Thanks for helping improve Grainy. This guide explains how to set up the project, make changes safely, and get code reviewed, merged, and deployed.

## Prerequisites

Install these before working on the project:

- Node.js `26.3.0` to match GitHub Actions.
- pnpm `11.0.7`. The repository pins this in `package.json`.
- Docker with Docker Compose for the local PostgreSQL database, migration runner, and database proxy in `dev/compose.yml`.
- Git.
- A Discord OAuth application if you need to exercise sign-in locally.
- A Cloudflare account and Wrangler access only if you need to preview or deploy the Cloudflare Worker.

Check your local versions:

```bash
node --version
pnpm --version
docker --version
```

## Repository overview

- `src/app` contains the Next.js App Router pages and API routes.
- `src/app/api/auth/[...all]/route.ts` exposes Better Auth handlers.
- `src/lib/auth` configures Better Auth and Discord OAuth.
- `src/lib/db` contains the Drizzle client and schema.
- `drizzle` contains generated SQL migrations.
- `dev/compose.yml` starts local PostgreSQL, runs migrations, and exposes the local database proxy.
- `wrangler.jsonc` configures the Cloudflare Worker generated by OpenNext.
- `.github/workflows/ci.yml` runs checks for pushes and pull requests.
- `.github/workflows/deploy.yml` deploys production from `main` after checks and migration handling.

## Local development setup

1. Clone the repository and enter it.

   ```bash
   git clone <repo-url>
   cd grainy
   ```

2. Install dependencies.

   ```bash
   pnpm install
   ```

3. Create a local environment file.

   ```bash
   cp .env.example .env.local
   ```

4. Fill in the values in `.env.local`. The default database and proxy values in `.env.example` match `dev/compose.yml`.

5. Start the local database, migration runner, and database proxy.

   ```bash
   docker compose -f dev/compose.yml up
   ```

   This starts:

   - PostgreSQL on `localhost:5432`.
   - A migration container that applies migrations from `drizzle`.
   - A database proxy on `http://localhost:8080` with token `token`.

6. Start Next.js in another terminal.

   ```bash
   pnpm dev
   ```

7. Open http://localhost:3000.

## Environment variables

Environment validation lives in `src/env.ts`. Local app development should use `.env.local`.

Do not commit real secrets. Keep local values in ignored files such as `.env.local` and `.env`.

### Required for the app

| Name                    | Required    | Purpose                                                                                                     | Local example                  |
| ----------------------- | ----------- | ----------------------------------------------------------------------------------------------------------- | ------------------------------ |
| `ENVIRONMENT`           | No          | Runtime environment, defaults to `development`. Valid values are `development` and `production`.            | `development`                  |
| `DRIZZLE_PROXY_URL`     | Yes         | HTTP endpoint for the Drizzle PostgreSQL proxy.                                                             | `http://localhost:8080`        |
| `DRIZZLE_PROXY_TOKEN`   | Yes         | Bearer token sent to the Drizzle proxy.                                                                     | `token`                        |
| `DISCORD_CLIENT_ID`     | Yes         | Discord OAuth client ID used by Better Auth.                                                                | Your Discord app client ID     |
| `DISCORD_CLIENT_SECRET` | Yes         | Discord OAuth client secret used by Better Auth.                                                            | Your Discord app client secret |
| `BETTER_AUTH_URL`       | Recommended | Base URL for Better Auth callbacks.                                                                         | `http://localhost:3000`        |
| `BETTER_AUTH_SECRET`    | Recommended | Secret used by Better Auth to sign and verify auth data. Use a long random value outside local development. | Random secret string           |

### Required for database CLI commands

| Name           | Required                 | Purpose                                                                                                       | Local example                                    |
| -------------- | ------------------------ | ------------------------------------------------------------------------------------------------------------- | ------------------------------------------------ |
| `DATABASE_URL` | Required for Drizzle Kit | PostgreSQL connection string used by `drizzle-kit`, including migration generation and migration application. | `postgres://grainy:grainy@localhost:5432/grainy` |

`DATABASE_URL` is optional for the running app because the app uses `DRIZZLE_PROXY_URL` and `DRIZZLE_PROXY_TOKEN` through `drizzle-orm/pg-proxy`.

### CI and deployment secrets

Production deployment uses GitHub Actions secrets and variables, not checked-in env files.

Cloudflare deployment requires:

- `CLOUDFLARE_API_TOKEN`
- `CLOUDFLARE_ACCOUNT_ID`

Migration detection and application require:

- `TS_OAUTH_CLIENT_ID`
- `TS_OAUTH_SECRET`
- `SSH_PRIVATE_KEY`
- `SSH_KNOWN_HOSTS`
- `SSH_HOST`
- `SSH_USER`
- GitHub Actions variable `INFRA_REPO_ROOT`

See `.github/CI_SETUP.md` before changing CI/CD secrets, environment protections, or workflow behavior.

## Common commands

| Command                      | Use                                                                        |
| ---------------------------- | -------------------------------------------------------------------------- |
| `pnpm dev`                   | Start the local Next.js development server.                                |
| `pnpm build`                 | Build the Next.js app.                                                     |
| `pnpm start`                 | Start a production Next.js server after `pnpm build`.                      |
| `pnpm lint`                  | Run ESLint.                                                                |

| `pnpm db:migration:generate` | Generate a Drizzle migration from schema changes. Requires `DATABASE_URL`. |

| `pnpm cf-typegen`            | Regenerate Cloudflare environment types in `cloudflare-env.d.ts`.          |
| `pnpm preview`               | Build and preview the app on the Cloudflare runtime locally.               |
| `pnpm upload`                | Build and upload through OpenNext/Cloudflare without deploying.            |
| `pnpm deploy`                | Build and deploy to Cloudflare Workers. Usually run by CI from `main`.     |

## Working with database changes

Database schema is defined in `src/lib/db/schema.ts`; generated migrations live in `drizzle`.

When changing the schema:

1. Update `src/lib/db/schema.ts`.
2. Make sure `DATABASE_URL` points at a safe local database.
3. Generate a migration.

   ```bash
   pnpm db:migration:generate
   ```

4. Review the generated SQL in `drizzle` before committing it.
5. Recreate or migrate your local database and run the app through the changed flow.
6. Include both schema and migration files in the pull request.

Do not edit already-applied production migrations unless you are deliberately correcting an unreleased migration before merge. Prefer adding a new migration for changes that may already exist in another environment.

## Code style

This project uses strict TypeScript, ESLint, and Prettier.

- Use TypeScript for application code.
- Prefer the `@/*` path alias for imports from `src`.
- Keep formatting controlled by Prettier. The config disables semicolons and sorts imports and Tailwind classes.
- Do not commit generated build output such as `.next`, `.open-next`, `.wrangler`, or `next-env.d.ts`.
- Regenerate `cloudflare-env.d.ts` with `pnpm cf-typegen` after changing `wrangler.jsonc` or Cloudflare bindings.
- Keep secrets out of source code, examples, screenshots, and logs.

## Before opening a pull request

Run the same checks CI runs:

```bash
pnpm format:check
pnpm lint
pnpm cf-typegen
pnpm exec opennextjs-cloudflare build
pnpm typecheck
```

Also run any relevant manual checks. For example:

- Start `docker compose -f dev/compose.yml up` and `pnpm dev` for local app changes.
- Visit http://localhost:3000 and exercise the affected pages or auth flow.
- Run `pnpm preview` when changing Cloudflare, OpenNext, headers, assets, or runtime behavior.
- Check generated migration SQL for database changes.

If you used `pnpm format:write`, review the diff before committing so unrelated formatting churn does not hide the real change.

## Pull request process

1. Create a focused branch from the latest `main`.
2. Keep the pull request scoped to one feature, bug fix, or maintenance change.
3. Write a clear PR description that includes:
   - What changed.
   - Why it changed.
   - How you tested it.
   - Any database migrations, new environment variables, or deployment considerations.
4. Link related issues or context when available.
5. Wait for CI to pass.
6. Request review from a maintainer or code owner.
7. Address review comments with follow-up commits.
8. Keep the branch up to date if required checks need to be rerun.

Recommended merge requirements for `main` are documented in `.github/CI_SETUP.md`: require a pull request, at least one approval, passing status checks, and up-to-date branches. If GitHub branch protection differs from that recommendation, follow the stricter rule.

## Review expectations

Reviewers should check:

- The change solves the stated problem without broad unrelated refactors.
- User-facing behavior was manually verified where applicable.
- Type safety is preserved without suppressions.
- Formatting, linting, typechecking, and OpenNext build checks pass.
- New or changed environment variables are documented.
- Database migrations are present, readable, and safe to apply.
- Cloudflare or Wrangler changes include regenerated types when needed.
- Secrets are not committed.

Authors should respond to every requested change. If you disagree with feedback, explain the tradeoff directly in the PR instead of silently ignoring it.

## CI process

`.github/workflows/ci.yml` runs on pushes and pull requests. It has three jobs:

- `format`: installs dependencies with `pnpm install --frozen-lockfile` and runs `pnpm format:check`.
- `lint`: installs dependencies and runs `pnpm lint`.
- `typecheck`: sets `SKIP_ENV_VALIDATION=true`, runs `pnpm cf-typegen`, builds OpenNext output, then runs `pnpm typecheck`.

The OpenNext build is part of typechecking because `cloudflare-env.d.ts` references `.open-next/worker`, which exists only after an OpenNext build.

## Deployment process

Production deployment is handled by `.github/workflows/deploy.yml` on pushes to `main`.

The deployment workflow:

1. Runs the same format, lint, and typecheck gates as CI.
2. Detects pending database migrations by connecting to the infrastructure host over Tailscale and SSH.
3. If migrations are pending, runs the `apply-migrations` job in the `apply-migrations` GitHub environment. That environment should require owner approval.
4. Deploys the Cloudflare Worker with OpenNext and Wrangler in the `deployment` GitHub environment.

The deploy job runs:

```bash
pnpm cf-typegen
pnpm run deploy
```

`pnpm run deploy` expands to:

```bash
opennextjs-cloudflare build && opennextjs-cloudflare deploy
```

Do not deploy production from a local machine unless a maintainer explicitly asks you to. Local deploys require Cloudflare credentials and can bypass the normal review, migration, and environment approval flow.

## Troubleshooting

### Environment validation fails locally

Check that `.env.local` exists and includes `DRIZZLE_PROXY_URL`, `DRIZZLE_PROXY_TOKEN`, `DISCORD_CLIENT_ID`, and `DISCORD_CLIENT_SECRET`. Use `.env.example` as the template.

### Drizzle commands fail with `DATABASE_URL is not defined`

`DATABASE_URL` is required for Drizzle Kit. Add it to `.env.local` or export it in your shell before running migration commands.

### The app cannot query the database

Make sure the local services are running:

```bash
docker compose -f dev/compose.yml up
```

Then confirm `.env.local` points at `DRIZZLE_PROXY_URL=http://localhost:8080` and `DRIZZLE_PROXY_TOKEN=token`.

### Typecheck cannot find `.open-next/worker`

Run the OpenNext build before typechecking:

```bash
pnpm exec opennextjs-cloudflare build
pnpm typecheck
```

### Wrangler or Cloudflare types look stale

Regenerate types:

```bash
pnpm cf-typegen
```

Commit `cloudflare-env.d.ts` if it changed.
